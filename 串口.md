====================
一、串口协议（已确定，必须按这个实现）
======================================

1. 单片机每一组发送 16 个字段。
2. 每个字段固定 4 个字节。
3. 每 4 个字节表示一个 IEEE754 单精度浮点数 float。
4. STM32 为小端序，所以串口收到的 4 字节顺序需要按小端 float 解释。
5. 例如串口收到：
   `4D BF B9 41`
   真实 float 字节序含义就是小端 float，对应数值约 `23.22`
   （也可以理解为人眼按大端看时要反过来成 `41 B9 BF 4D`，程序里直接按 `<f` 解包即可）
6. 一整组数据固定长度：
   - `16 * 4 = 64 bytes`
7. 不要再按旧版的字符串数组、正则提取数字、`[1,2,3]` 文本协议来处理。
8. 新版程序必须以“原始二进制字节流”方式从串口读取，并在缓冲区中自行组帧。

====================
五、组帧与串口读取要求（这是重点）
==================================

我现在不确定组与组之间的时间间隔是多少，所以程序不能靠固定 sleep、固定行结束符来判断一组完成。

必须按下面思路实现：

1. 串口读到的是连续字节流，先进入 byte buffer
2. 一组完整数据的唯一硬条件是：缓冲区中拿到了连续 64 字节
3. 程序必须有“组帧器 / frame assembler”逻辑，例如：
   - `SerialFrameAssembler`
   - 内部维护 `bytearray buffer`
   - 持续 append 新读到的字节
   - 当 `len(buffer) >= 64` 时尝试取出一组
4. 不要依赖固定的“组间时间间隔”
5. 但可以使用“字节间空闲时间”作为辅助优化，例如：
   - 如果超过某个 gap 没收到新字节，且缓冲区不足 64 字节，可以暂时等待
   - 如果缓冲区里有脏数据长期无法组成合法帧，则清理/重同步
6. 因为没有明确帧头帧尾，所以必须做合法性校验来辅助判断当前 64 字节是不是有效帧

====================
七、重同步策略（必须实现）
==========================

因为串口流可能从中间开始、可能丢字节，也可能粘包，所以必须有重同步逻辑。

建议实现：

1. 主路径先按 64 字节整帧读取
2. 若解析失败：
   - 优先尝试按 4 字节步长滑动重试（因为字段是 float）
   - 如果连续多次失败，再允许按 1 字节滑动寻找下一组可能的起点
3. 找到第一组合法帧后，再恢复正常 64 字节节拍读取
4. 记录日志：
   - 当前 buffer 长度
   - 丢弃了多少字节
   - 何时重新同步成功

目标：

- 不完整的一组数据不能上传
- 下一组开始时，程序能自动恢复，而不是卡死或持续错位
