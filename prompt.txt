C. 后端（FastAPI）需求
========================
1) 技术栈：
   - FastAPI async + uvicorn
   - SQLAlchemy 2.x async + asyncpg
   - Alembic migrations
2) API（路径固定）：
   - POST /api/v1/runs/start   body:{machine_id} -> {run_id, run_name, started_at}
   - POST /api/v1/runs/{run_id}/stop -> {run_id, ended_at}
   - GET  /api/v1/runs?days=30&machine_id=... -> runs[]
   - GET  /api/v1/runs/{run_id}/gps?limit=100000 -> gps_points[]
   - GET  /api/v1/runs/{run_id}/telemetry?from=&to=&bucket=1s -> 聚合后的序列
   - POST /api/v1/ingest  Header Authorization: Bearer INGEST_TOKEN
       Body 标准 JSON：
       {
         "ts":"ISO8601",
         "machine_id":"string",
         "telemetry":{ seed_channels_g:[5], seed_total_g, distance_m, leak_distance_m, speed_kmh, alarm_blocked, alarm_no_seed },
         "gps":{ lon, lat, alt_m?, heading_deg? }
       }
       允许 telemetry 或 gps 缺失。
       行为：必须写入“该 machine_id 当前 active run”。若不存在 active run，返回 409。
3) WebSocket：
   - /ws/live?run_id=...
   - 后端维护 run_id -> clients 的广播表
   - ingest 入队后由 broadcaster 按频率（telemetry 5Hz、gps 1Hz）推送最新值给订阅该 run_id 的客户端
4) 写库与自适应（必须实现）：
   - ingest 接口只做校验与快速入队，返回 202
   - asyncio.Queue + worker 批量写库：
       DB_BATCH_MAX_SIZE 或 DB_BATCH_MAX_LATENCY_MS 触发一次 batch insert
   - 队列超过 QUEUE_HIGH_WATERMARK 时启用 aggregate 背压：
       telemetry 合并为“最后值”，gps 只保留每秒最后一条
5) 数据库表（固定表，禁止动态建表）：
   runs, telemetry_samples, gps_points（含外键 ON DELETE CASCADE），并建 (run_id, ts) 索引
6) 30 天保留：
   - 后端启动时创建后台任务，每24h执行一次：
       DELETE FROM runs WHERE started_at < now() - interval '30 days'
     级联删除样本数据
7) docker-compose：
   - db 使用 timescale/timescaledb:latest-pg16（或等价），暴露 5432
